## What this app does

TechOps Crosscheck is a verification tool for change initiatives. A user picks a
pivot date (a migration cutover, a vendor transition, a platform launch, any event
with a clear before-and-after) and a symmetric window (30, 60, 90, or 120 days on
each side). The app compares operational health pre vs post pivot across United's
TechOps application portfolio, surfacing wins and regressions in a single view,
with estimated revenue impact in dollars.

The app is the productized version of an existing Dynatrace dashboard
(TechOps Pivot Comparison, version 21 in source control). The source dashboard's
DQL is the starting point for every query in this app; query files note the
provenance.

The target audience is United TechOps and engineering leadership who need to
quantify the operational impact of a change. The verification register matters:
this is Crosscheck, not Diagnose. The app answers "did the change deliver what
we said it would" with numbers.

## Architecture

### Page structure

Single-page React app with three primary regions and one secondary region.

Replace the existing scaffold pages:
- ui/app/pages/Home.tsx becomes the Crosscheck dashboard (the main view).
- ui/app/pages/Data.tsx becomes the Saved Pivots manager.

Header (ui/app/components/Header.tsx) keeps its role as nav chrome. Update nav links to read "Crosscheck" (Home) and "Saved Pivots" (Data). Remove the "Data" label.

Card (ui/app/components/Card.tsx) is the building block. Extend it (or wrap it) for the metric-card and tier-card variants used in the headline and tier rollup regions.

Within the Crosscheck page, top to bottom:

1. Control bar: pivot date (date picker), window days (segment selector: 30, 60, 90, 120), tier filter (multi-select), director filter (multi-select), application multi-select, saved pivots dropdown, source health indicator. The date picker, segment selector, and multi-selects come from Strato. Use strato_search to find the right components.

2. Headline strip: four metric cards in a 4-column grid. Estimated revenue impact, Problem count, Weighted average MTTR, Portfolio verdict. Each card shows pre value, arrow, post value, percent change with up/down indicator, and a sparkline below covering the full pre+post window with a dashed vertical pivot line. Sparkline pre is blue, post is red, with translucent area fills.

3. Tier rollup row: four tier cards in a 4-column grid (Tier 1, 2, 3, 4). Each tier card shows three diverging mini-bars (MTTR percent change, Count percent change, Impact percent change) with the percent value on the right.

4. Per-CI table: the main visualization. Each row is one ApplicationCI. Columns: ApplicationCI name and full app name, Tier badge, Director, MTTR percent change diverging bar, Count percent change diverging bar, Impact percent change diverging bar, Revenue impact twin-bar (pre and post in dollars). Sortable by every column. Filterable via the control bar plus a verdict toggle (All, Regressed, Improved, Newly emerged). Bars clip at plus or minus 300 percent for readability with a chevron marker for clipped rows.

5. Saved Pivots panel (collapsible or on its own page at /saved-pivots): list of saved pivots with name, date, window, optional annotation. Each entry has a button to load that pivot into the main view. Add, edit, delete actions on each row.

### Drilldowns

- Click an ApplicationCI row: opens a Sheet (Strato component) with the per-problem table for that CI, scoped to either pre or post window via a toggle. Each problem row links to the Davis Problems app at https://ual.apps.dynatrace.com/ui/apps/dynatrace.davis.problems/problem/<event.id>.
- Click a tier card: filters the per-CI list to that tier.
- Click the verdict card: filters to either Regressed or Improved (the verdict the card currently displays).
- Click a director name: filters to that director's portfolio.

### Reusable components

Factor these as standalone components under ui/app/components/ so they can be lifted into other United-internal Dynatrace apps later:

- DivergingBar: takes a percent value, renders a centered-zero bar with green-left-red-right semantics, handles the plus or minus 300 percent clipping with a chevron mark, handles the special cases (newly emerged shows "new" badge, fully resolved shows "resolved" badge).
- Sparkline: takes an array of numeric values and a pivot index, renders an SVG sparkline with the dashed pivot line and the pre/post color split.
- MetricCard: takes a label, pre value, post value, formatter function, sparkline data, and renders the headline metric card.
- TierCard: takes a tier label, CI count, and three percent-change values, renders the tier rollup card.
- TwinBar: takes pre and post numeric values and a formatter, renders the pre/post comparison bars with values to the right.

Co-locate these with their tests if you write tests. Tests are not required for v1 but the components should be written to be testable (pure props, no side effects).

### Reference data: TechOps application list

The TechOps application universe is sourced first from `load "/lookups/techops/techopsApplicationList"`. The lookup schema is AppCI, ApplicationName, Tier, Director.

Resolution order at app init:

1. Try the central lookup query.
2. If it returns rows, use those.
3. If it returns zero rows, prompt the user with the upload offer.
4. If it errors with a permission error, prompt the user with the upload offer (see Permission failure UX below for the formal handling) AND surface the upload-table fallback.
5. If the user has uploaded a table (stored in user app state under the key "techops:applicationList:uploaded"), use that table.

A persistent banner at the top of the app indicates the active source: "Using central TechOps lookup (N rows, last loaded HH:MM)" or "Using uploaded table (N rows, uploaded YYYY-MM-DD)". The banner has an action button: "Switch to uploaded table" or "Switch to central lookup", whichever is the inactive option.

When central access becomes available later for a user who has been on uploaded fallback, the banner shows: "Central lookup is now available with N rows. Switch?" with explicit accept/decline buttons. Do not auto-switch.

### Upload TechOps table modal

Triggered from the banner action or from the empty state when no source is available. A Strato Modal (use strato_search to find the component). Accepts CSV or pasted TSV. Validates required columns (AppCI, ApplicationName, Tier, Director). Shows a preview of the first ten parsed rows. Flags duplicate AppCI values and missing required values. Normalizes Tier values to the canonical four strings:

- "1 - most critical"
- "2 - somewhat critical"
- "3 - less critical"
- "4 - not critical"

Accept variants ("1", "T1", "Tier 1", "tier1") and map them. Reject anything else with an inline message. Director and ApplicationName are free text. AppCI must be non-empty and unique within the table.

On save, write the table to user app state under "techops:applicationList:uploaded" with a JSON payload of {rows: [...], uploadedAt: ISO timestamp, uploadedBy: current user identity from app-environment if available}. The app re-initializes against the new source without a page reload.

### Permission failure UX

Every Grail or settings call routes through a centralized error handler that distinguishes permission errors (HTTP 401, 403, or specific missing-scope responses from the SDK) from operational errors (timeouts, query failures, malformed results).

On a permission error, render a "Permission required" state. The state shows:

1. A user-facing reason: one sentence, plain English. Example: "TechOps Crosscheck needs access to Dynatrace business events to load financial impact data."
2. A code block showing the specific scope that is missing. Example: storage:bizevents:read. The code block is selectable and copy-friendly so the user can paste it into a ticket.
3. A "Contact United Observability Team" button that opens a mailto: link to observability@united.com with:
   - Subject: TechOps Crosscheck access request
   - Body: a pre-filled message containing the missing scope, the user's identity if available from @dynatrace-sdk/app-environment, the tenant URL (https://ual.apps.dynatrace.com), and a UTC timestamp.
4. A Retry button alongside in case the failure was transient.

The handler triggers in three places with three different surface levels:

- App init (lookup or settings fail at startup): full-page Permission required state, no other UI rendered.
- First query run (headline cards, tier rollup, per-CI table): card-level Permission required state, replacing just the affected region.
- Drilldown (per-problem detail in the Sheet): inline Permission required state inside the Sheet.

The same handler also covers the upload-table fallback path: if the user lacks both lookup access and state:user-app-states:write for saving the uploaded table, the empty state shows a Permission required full-page state instead of the upload CTA.

## Queries (lifted from the source dashboard, parameterized for the app)

All queries live as named exports in ui/queries/<name>.ts. Each module exports a function that takes the parameters it needs and returns a DQL string. This keeps DQL inspectable and copy-pasteable into a Notebook for debugging.

Use dql_search to verify any unfamiliar DQL syntax before writing the queries. Some non-obvious gotchas already known from working with this dataset:

- DQL does not accept the long literal suffix L. Use plain integer literals (90) or toLong(90).
- duration() requires the unit string "h" (or "m", "s", "d"). Pivot math: `pivot - duration(24 * windowDays, "h")` for the pre-window start.
- dt.davis.problems is event-storage. resolved_problem_duration is in nanoseconds. Divide by 86,400,000,000,000 (8.64e13) for days.
- ApplicationCI extraction from entity_tags is non-trivial. The pattern is in the source dashboard and should be lifted verbatim.
- summarize with multiple aggregations plus a by clause requires curly-brace wrapping of the aggregation list.

The queries:

### ui/queries/techopsApplicationList.ts

Loads the TechOps application taxonomy. Cached at app init with a long staleTime via useDql. No parameters.

```dql
load "/lookups/techops/techopsApplicationList"
| filterOut AppCI == "AppCI"
| fields AppCI, ApplicationName, Tier, Director
```

### ui/queries/perCiRollup.ts

The main per-CI comparison query. Parameters: pivotDate (ISO date string), windowDays (number), tierFilter (string array, optional), directorFilter (string array, optional), appCiFilter (string array, optional). Returns one row per ApplicationCI with pre and post MTTR, count, impact, plus the three percent-change values.

This query is lifted from tile 5 of the source dashboard with the limit and sort adjusted (60 rows is the source default; we keep 60 but expose a way to remove the limit later).

### ui/queries/tierRollup.ts

Aggregate query grouped by Tier. Parameters: same as perCiRollup. Returns four rows (one per tier) with pre/post MTTR, count, impact totals.

This is the same shape as the second query I ran in the data exploration phase. Lift it from there.

### ui/queries/dailyTimeseries.ts

Daily timeseries over the full pre+post window for sparklines. Parameters: pivotDate, windowDays. Returns one row per day with problems count, average MTTR, and total impact.

This is the same query I ran for the sparkline data in the v2 widget. Lift it from there.

### ui/queries/problemDetailPre.ts and ui/queries/problemDetailPost.ts

Per-problem detail for a single ApplicationCI in the pre or post window. Parameters: pivotDate, windowDays, ciId (string). Returns one row per problem with display_id, event.id (for the link), event.name, event.status, dt.davis.impact_level, resolved_problem_duration, dt.davis.affected_users_count, timestamp.

These are lifted from tiles 6 and 7 of the source dashboard.

### Bizevents financial loss lookup

Embedded as a sub-query (lookup [...]) inside perCiRollup, tierRollup, and dailyTimeseries. Lifted as-is from the source dashboard. Do not re-implement; use the exact pattern from tile 5.

- Team-shared saved pivots (Document Service integration).
- Write actions of any kind to ServiceNow or to the central techopsApplicationList lookup.

## Known gotchas

### Stuck Davis problems

The current production data contains a small number of CIs with very long-duration unresolved Davis problems (ASG, TRIDIUM, DYP at the time of writing). These produce post-MTTR values of weeks or months that distort the sparkline scale and the percent-change calculation.

The app does not filter these out automatically in v1. The diverging bar's plus or minus 300 percent clip prevents them from compressing the rest of the rows visually. A "data quality" filter that hides CIs with post-MTTR longer than the window length is a candidate for v2 and should not be implemented in v1.

If a user asks why a particular CI shows an extreme value, the per-problem drilldown is the correct answer: it shows the underlying problem with its actual duration, and the user can investigate from there.

### Newly emerged CIs and zero-post-count CIs

A CI with zero pre-window problems and N post-window problems is "newly emerged" (preCount = 0). The percent-change calculation cannot produce a meaningful number, so the app uses the sentinel value 9999 in the data and renders "new" in the UI.

A CI with N pre-window problems and zero post-window problems is a genuine improvement: postCount = 0 means no problems were detected in the post window, not that the application was "resolved" in any special sense. The percent change is -100. The app renders this as a regular -100% improvement (a full-width green diverging bar). These rows appear under the "Improved" verdict filter.

A CI with zero pre-window problems and zero post-window problems had no issues detected in either window. All three pct-change fields are null, so the diverging bars render "—". These rows appear in the "All" view but not in any verdict-specific filter.

The verdict toggle has a "Newly emerged" filter that selects rows with the 9999 sentinel. There is no separate filter for zero-post-count rows; they appear under "Improved".

### dt-app Node version

The dt-app CLI requires Node 24. Verify the user's Node version before running dt-app commands. If they're on an older version, surface the warning and stop rather than letting the command fail confusingly.

### Vendor prefix in app ID

The app ID my.tech.ops.crosscheck has the default vendor prefix "my". This is fine for development. Before any production deployment, the prefix should be changed to a United-internal vendor prefix (suggested: united). Do not hardcode the app ID anywhere outside app.config.json. If you need it at runtime, read it from @dynatrace-sdk/app-environment.

## Style notes

- All numbers displayed in the UI must go through the formatters in app/lib/formatters.ts. Float math leaks artifacts; round explicitly.
- Use Strato design tokens for all colors, spacing, and typography. Do not hardcode hex values for state colors (success, danger, warning) anywhere in components.
- Diverging bars and sparklines are the exception: they use specific brand-aligned color values (green for improvement, red for regression, blue for the pre series). Document those values in app/lib/colors.ts so they are centrally maintained.
- Match the source dashboard's terminology in column headers and labels: ApplicationCI, MTTR, Tier, Director.
- The banner copy, the upload modal, and the permission failure state should all use sentence case.
